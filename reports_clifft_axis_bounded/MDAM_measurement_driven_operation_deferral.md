# MDAM Measurement-Driven Operation Deferral

**Status: IMPLEMENTED, default-OFF, authoritative paths preserved, bit-identical records on all
benchmarks/paths.**  Feature flag `CliftAxisBoundedNearClifford._measurement_tail_deferral` (default
`False`).  Backup tag `mdam-before-measurement-tail-deferral-20260626_170315` (+ working-tree patch in
`artifacts/mdam_measurement_tail_deferral/`).

Every claim below is **CODE-PROVEN** (cited lines) or **MEASURED** (cited artifact). The one
exactness *condition* and its scope are stated explicitly.

## 1. Motivation

A measurement core is the anticommutation transitive closure of the measured physical Pauli `Z_q`
over the pending ledger ([lazy.py `_core_indices`](../nearclifford_backend/lazy.py)). Diagonal
(`I/Z`-on-`q`) rotations enter the core only via the closure (coupled to a branch-mixing seed on
another axis), yet the baseline executes the **entire** ordered core on the `2^(r_out+1)` joint —
including a trailing suffix of rotations that **commute with the measurement** and therefore cannot
affect this measurement's Born probability or outcome.

This patch splits the ordered core into an immediate-execution **prefix** and the maximal trailing
measurement-commuting **suffix** (the *tail*); only the prefix is executed/consumed at this
measurement. The tail is record-irrelevant to the current measurement; whether it is ever needed
again is decided by the existing structure machinery.

```
baseline:  core = immediate dense execution set
patch:     core = immediate execution prefix  +  deferred commuting suffix
```

## 2. Existing semantics (CODE-PROVEN)

- Pending entry `[x, z, phase, theta, uid]`, insertion-ordered dict, physical-qubit Pauli basis,
  conjugated per Clifford ([lazy.py:111-134](../nearclifford_backend/lazy.py)), uid monotonic.
- Core selection = anticommutation transitive closure of `(qx,qz)=(0,1<<q)` over pending
  ([lazy.py `_core_indices`](../nearclifford_backend/lazy.py)). `_commute_xz(qx,qz,x,z)` is the
  physical-basis commute test.
- Three executor paths, all selecting the **same** ordered core via `_fused_core_entries(q)`:
  compiled C++ (`compiled_core.try_compiled_measure`), fused symbolic
  (`bounded._fused_setup`/`_fused_commit`), oracle dense (`lazy._flush_core`/`_do_flush`).
- **Dead-rotation drop** (`drop_dead`, default ON, [backend.py:120-130](../nearclifford_backend/backend.py)):
  the structure pre-pass records `_flushed_uids`; uids **never flushed** in any core are pruned at
  creation by `apply_rotation` ([lazy.py:140-141](../nearclifford_backend/lazy.py)) — record-exact.

## 3. Mathematical exactness — and its precise condition

Write the ordered core unitary `U_full = U_T U_C` (`U_C` prefix, `U_T` = trailing
measurement-commuting suffix). Each tail rotation `R_P(θ)` commutes with the measurement projector
`Π_b` iff `P` commutes with `Z_q` iff `P` has no `X` on `q` — exactly `_commute_xz(0,1<<q,x,z)`.
Hence `[Π_b, U_T] = 0` and `Π_b U_T U_C |ψ> = U_T Π_b U_C |ψ>`: projecting before vs after the tail
is identical, and (`U_T` unitary) the surviving branch norm is unchanged, so the **normalised**
survivor is identical.

**Critical condition (DISCOVERED, then proven):** the equality `Π_b U_T U_C = U_T Π_b U_C` requires
`U_T` to be **unitary**, i.e. every deferred tail rotation must be generated by a **Hermitian** Pauli.
The engine represents a per-qubit `(x=1,z=1)` slot as `m = X@Z = -iY` (anti-Hermitian); a Pauli with
an **odd number of Y-slots** at `phase=0` is anti-Hermitian, making `R = cos - i·sin·(X@Z)`
**non-unitary**. For a non-unitary "rotation," normalise-then-apply ≠ apply-then-normalise, so
untouched-tail deferral is **NOT** exact. A Hermitian Pauli has `phase ≡ popcount(x&z) (mod 2)`.

This is not a real limitation: **every real MDAM pending entry and every deferred tail is Hermitian**
(verified §4). The anti-Hermitian failures appeared only when a synthetic test fed `phase=0` odd-Y
generators through `apply_rotation` (a non-physical, non-unitary "rotation").

## 4. Untouched-tail proof (Phase-1 gate, MEASURED)

Probe `/tmp/p1_exactness_probe.py`; artifact `exactness_probe.json`. The candidate reuses the
**authoritative oracle measure_z**, monkeypatching only core *selection* to return the prefix,
leaving the tail untouched, then flushing it later — forced outcomes (both keepbit).

- **Structured 20/20 PASS** (aligned |Δstate| ≤ 2.8e-16, overlap 1.0): incl. `Z_q⊗Z_q1` extra-support,
  `tail_len=2`, mid-kept `Z;X;Z`, off-diagonal `Z_q⊗X_q1`/`Z_q⊗Y_q1`, and **mutually-anticommuting
  off-diagonal pairs**.
- **Random sweep 2000/2000 PASS** (n=2..8, both keepbit), **worst aligned |Δ| = 5.55e-16**.
- **Independent dense ground truth** (`/tmp/p1_groundtruth.py`): baseline matched ground truth at
  1e-16 throughout; only the artificial non-unitary candidate diverged (root-caused to anti-Hermitian
  generators, §3).
- **Real-workload condition check** (`/tmp/p1_hermitian_invariant.py`; `hermitian_invariant.json`):
  **0 Hermitian-invariant violations** across 9 benchmarks × 3 seeds — every pending entry AND every
  deferred tail satisfies `phase ≡ popcount(x&z) (mod 2)`. Real tails are in fact **pure-Z**
  (0 Y-slots). The Phase-1 exactness condition therefore always holds in practice.

The production-flag path (not the prototype) was independently re-verified: `/tmp/td_state_exactness.py`
(`state_comparison_results.json`) — **structured 14/14 + random sweep 1200/1200 PASS**
(worst 4.5e-16), with tails retained untouched in pending.

## 6/7/8. Compiled / Fused / Oracle paths

All three select the same core via `_fused_core_entries(q)`, run it through
`_measurement_execution_plan`, execute and consume only `execute_entries`, and never touch the tail.
The C++ executor ABI is unchanged (it merely receives a shorter `rots`). Compiled eligibility-failure
fallback to the oracle is preserved (plan creation is read-only; pending is consumed only after the
executor succeeds; the oracle re-derives the same plan). **Record bit-identity verified on all three
paths** (§13).

## 9. Pending / UID semantics

The split preserves every tail entry's uid, x/z masks, phase, theta, and relative order; no entry is
re-inserted or re-uid'd; only `execute_uids` are `del`'d from pending. Verified by the tail-retention
checks (`exactness_probe.json`, `state_comparison_results.json`: `tail retained & untouched = True`
on every config).

## 10. Structure-cache regeneration (Phase 6)

The structure pre-pass ([backend.py `_structure_for`](../nearclifford_backend/backend.py)) runs the
DYNAMIC path under the flag, so `_record_cores` (→ `_fast_cores`) stores the **execute-prefix** uids,
and `dead = all uids − flushed` now includes the deferred tails. Real shots gather the recorded
prefix from the fast table (no live scan). `_fast_cores` is therefore the **flag-keyed** schedule by
construction (it is rebuilt by discovery under whichever flag value is set before the backend runs;
`_struct_cache` is per-backend-instance keyed by `id(prog)`). Cross-check available via
`_debug_compare` (fast prefix == dynamic prefix). The validation in §4 (fast disabled / synthetic)
and §13 (fast enabled / real) both pass.

## 11. RNG and outcome invariants

Plan creation and the split consume **no RNG** (pure read-only classification). Record bit-identity
ON==OFF (§13) implies the RNG draw sequence, internal outcomes, and record bits are identical.

## 14. Tail lifecycle (MEASURED) — the key real-backend finding

`/tmp/td_savings.py`; `tail_execution_rank.csv`, `savings_summary.json`. In the **real backend**
(`drop_dead=True`, default) the deferred tails are record-irrelevant and **never flushed**, so the
existing dead-drop **eliminates** them at creation (the cores shrink by exactly the tail count). The
runtime split therefore peels nothing (the tails are already gone); the deferral's effect is realised
entirely through the discovery pass marking the tails dead.

| benchmark | flushed OFF→ON | reduction | tail fate |
|---|---|---|---|
| coherent_d5_r5 | 1209 → 1065 | **144 (11.9%)** | permanently dead (eliminated) |
| coherent_d3_r3 | 195 → 171 | **24 (12.3%)** | permanently dead (eliminated) |
| cultivation_d5 / coherent_ry_d3_r3 / distillation | — | 0 (no tail) | — |

(Matches the directive's §17.3 prediction 1209→1065 / 195→171 exactly.) With `drop_dead=False`
("pure deferral") the tails instead stay UNTOUCHED in pending to shot end (95/shot d5_r5, 24/shot
d3_r3) — **all permanently dead** (none re-enters a later flushed core: the tails are pure-Z on the
already-measured qubit). Both regimes are record bit-identical.

## 15. FLOP / touch changes (MEASURED)

The eliminated rotations are the **trailing diagonal (pure-Z, measured-axis)** rotations — the cheapest
kind (strided half-array diagonal apply, no qubit promotion). The reduction is **11.9 % / 12.3 %** of
flushed rotations on the two tail benchmarks, **0 %** on the controls (negative control: flag-ON
overhead ≈ 0).

## 16. Workspace / rank changes (MEASURED) — INVARIANT HELD

`savings_summary.json`: **max peak-resident Δ = 0, max maxM Δ = 0** across all benchmarks/seeds.
Tail deferral causes **no rank/workspace increase** (and no decrease either): the eliminated tails are
diagonal on the measured axis, so they never promoted a qubit and were never the peak driver. **The
win is FLOP-only, not peak-memory** — for these QEC benchmarks (pure-Z tails) there is no peak win.

## 17. Wall-time (MEASURED)

`/tmp/td_unit_wall.py`; `walltime_comparison.csv` (compiled path, median of 7, warmed). With the
flag-keyed fast schedule (§10) and the verification assertions gated off the hot path, tail deferral
is **wall-neutral**: coherent_d5_r5 ≈1.01×, coherent_d3_r3 ≈1.01×, controls within ±3 % noise.
Isolation (`/tmp/…`): the deferral's *own* effect (ON-dyn vs OFF-dyn) is 1.005× / 0.987× ≈ neutral;
the only material cost was the live-scan overhead from disabling `_fast_cores`, which Phase 6 removes.
**There is no wall speedup** — the eliminated rotations are too cheap and peak is unchanged.

## 18. State-sharing compatibility

The plan cleanly separates `execute_uids` from `deferred_tail_uids`; deferred tails do not participate
in the current Born/survivor. No large state-sharing refactor was attempted (out of scope). The phi /
frame / pending remain owned in-place by one `nc`.

## 19. Fallbacks and unresolved risks / honest scope

- **Correctness is record-exact, not state-exact, on real benchmarks.** With `drop_dead` (default ON)
  the eliminated tails are NOT applied to the final dense state, so flag-ON `statevector()` differs
  from a hypothetical tail-flushing baseline — exactly the same record-exact-not-state-exact contract
  the existing `drop_dead` already provides. Synthetic state-exactness (no dead-drop) is machine
  precision (§4). Real-benchmark **records are bit-identical**.
- **No peak/FLOP win materialises for the current QEC benchmarks** because their tails are pure-Z
  diagonal on the measured axis. A real win (peak reduction) would require tails with off-diagonal /
  promoting support — none of the shipped benchmarks have these. This is the honest ceiling of the
  optimisation on this workload, not an implementation defect.
- Fallbacks: empty-prefix → full execution; structural fast-table miss → dynamic+split; compiled
  eligibility failure → oracle (pending untouched until executor success). All record-exact.
- Phase-1 exactness holds **only for Hermitian tails**; verified universal on real workloads (§4). A
  future circuit emitting an anti-Hermitian (non-unitary) pending rotation would violate the
  *simulator's* unitarity assumption independent of this patch; the `_tail_defer_assert` debug guard
  cross-checks the split classification.
